## high: Default DatabaseSeeder can create known super-admin credentials outside dev data setup

id: fnd_sig-feat-infra-3190b8194b-0bac36_ed90f0705f
category: security
confidence: high
triage: confirmed-bug
status: open
feature: Laravel seeders database/seeders (feat_infra_3190b8194b)

evidence:
- database/seeders/DatabaseSeeder.php:14-17 (DatabaseSeeder::run)
- database/seeders/FamilyMemberSeeder.php:63-77 (FamilyMemberSeeder::run)
- database/seeders/FamilyMemberSeeder.php:84-123 (FamilyMemberSeeder::run)

The default db:seed path calls FamilyMemberSeeder, which falls back to public email/password values and creates a super-admin plus demo users. If the default seeder is run against any non-isolated environment without explicit admin config, the application gets a known privileged login.

recommendation:
Gate demo seeding to local/testing environments or require explicit credentials before creating privileged users. Keep production bootstrapping on ProductionSeeder and make DatabaseSeeder refuse to create known-password accounts outside safe environments.

test analysis:
ProductionSeederTest verifies ProductionSeeder avoids demo contribution/payment data, but no test runs DatabaseSeeder under a production-like environment or asserts that the known fallback admin is blocked.

suggested regression test:
Add a test that sets the environment to production, runs DatabaseSeeder, and asserts admin@family.test is not created and the seeder fails or skips demo users explicitly.

minimum fix scope:
Add an environment/config guard around DatabaseSeeder or FamilyMemberSeeder and cover it with one seeder safety test.

repro:
Run php artisan db:seed in an environment without ADMIN_EMAIL and ADMIN_PASSWORD configured. DatabaseSeeder calls FamilyMemberSeeder, which creates admin@family.test with password password and is_super_admin true.

## medium: Concurrent command runs can send duplicate reminders before the sent-at flag is written

id: fnd_sig-feat-cli-command-1169cff156-_c436e621af
category: concurrency
confidence: high
triage: confirmed-bug
status: open
feature: Laravel command contributions:remind (feat_cli-command_1169cff156)

evidence:
- app/Console/Commands/SendContributionReminders.php:32-39 (SendContributionReminders::handle)
- app/Console/Commands/SendContributionReminders.php:45-46 (SendContributionReminders::handle)
- tests/Feature/Commands/SendContributionRemindersTest.php:45-60

The idempotency check is a non-atomic read of a nullable column, and the flag is written only after notification dispatch. If two processes run the command at the same time, both can select the same contribution while the sent-at column is still null, both dispatch the notification, and then both save the timestamp. The existing duplicate test only covers sequential reruns after the first process has already saved the flag, so it does not prove the concurrent case safe.

recommendation:
Claim each contribution atomically before dispatching, for example by updating the sent-at column with a `whereKey($id)->whereNull($sentAtColumn)` condition and only notifying when that update affects one row, or by using an appropriate row lock/transaction around the claim step.

test analysis:
The test suite checks that a second run after the first has completed sends zero reminders, but it does not simulate two command instances selecting the same rows before either writes the sent-at timestamp.

suggested regression test:
Add a test that simulates a stale selected contribution by setting the sent-at column between selection and dispatch, then asserts the command does not notify unless an atomic claim update succeeds.

minimum fix scope:
Change `SendContributionReminders::handle` so notification dispatch is gated by an atomic sent-at claim for the selected contribution.

repro:
Start two `contributions:remind --day=25` processes against the same unpaid contribution at the same time; both can pass the `whereNull(reminder_sent_at)` query before either reaches the save at line 46.

## medium: Invalid month option can generate corrupt period data

id: fnd_sig-feat-cli-command-a960e40f9a-_523d25dc66
category: bug
confidence: high
triage: confirmed-bug
status: open
feature: Laravel command contributions:generate (feat_cli-command_a960e40f9a)

evidence:
- app/Console/Commands/GenerateMonthlyContributions.php:18-24 (GenerateMonthlyContributions::handle)
- app/Console/Commands/GenerateMonthlyContributions.php:57-65 (GenerateMonthlyContributions::handle)

The signature says --month must be 1-12, but handle() casts arbitrary input and passes it to Carbon. Carbon can normalize overflow values such as 13 into a different calendar month while the raw invalid month is still persisted to contributions.month. That can create rows like year=2026, month=13 with a due_date in January 2027, breaking period identity and later queries that assume months are 1-12.

recommendation:
Validate month and year before any date construction or database writes. Reject month values outside 1-12 with a clear error and non-success exit code.

test analysis:
No linked tests were provided, and there is no included test asserting invalid command options are rejected.

suggested regression test:
Add a Pest console test that runs contributions:generate with --month=13 and asserts a failure exit code and zero created Contribution rows.

minimum fix scope:
Add option validation in GenerateMonthlyContributions::handle and cover invalid date options with a focused console test.

repro:
Run php artisan contributions:generate --year=2026 --month=13 for a family with an active categorized member, then inspect the generated Contribution row.

## medium: Month-end seeding can create duplicate current-month contributions

id: fnd_sig-feat-infra-3190b8194b-fe873d_bd90df2c1f
category: bug
confidence: high
triage: confirmed-bug
status: open
feature: Laravel seeders database/seeders (feat_infra_3190b8194b)

evidence:
- database/seeders/FamilyMemberSeeder.php:127-129 (FamilyMemberSeeder::run)
- database/seeders/FamilyMemberSeeder.php:135-160 (FamilyMemberSeeder::run)

Carbon's default month overflow semantics mean dates like 2026-05-31 subtract one month to 2026-05-01, not 2026-04-30. On those dates, the seeder treats the previous contribution and current contribution as the same year/month for every member. That either creates corrupt demo data or trips the contribution uniqueness constraint for user/month.

recommendation:
Compute the previous period without month overflow, for example from the first day of the current month or with subMonthNoOverflow(), and use fresh immutable/copy instances for due_date and paid_at values.

test analysis:
No tests are listed for FamilyMemberSeeder. The existing ProductionSeederTest exercises ProductionSeeder only and does not freeze time around month-end dates.

suggested regression test:
Add a Pest test that travels to 2026-05-31, runs FamilyMemberSeeder, and asserts each paying member has one previous-month contribution for April 2026 and one current-month contribution for May 2026 with no duplicate-key failure.

minimum fix scope:
Update FamilyMemberSeeder previous-month date calculation and add one date-boundary seeder test.

repro:
Freeze time to 2026-05-31 and run FamilyMemberSeeder on an empty migrated database. The first member gets a May previous contribution, then the current contribution also targets May.

- database/migrations/2026_04_05_131950_create_agent_conversations_table.php:25-34

The migration stores AI conversation content but leaves both user_id columns unconstrained and leaves message conversation_id as an index only. suggested regression test:
Add a feature test that seeds an AI conversation/message for a user, deletes the account through profile.destroy, and asserts the configured retention behavior: rows are deleted, anonymized, or user_id is nulled.

minimum fix scope:
Forward migration for AI conversation referential integrity or explicit account-deletion cleanup plus a profile deletion regression test.

repro:
Create a user with an AI conversation and message, then delete the user account. The user row is gone, but the agent_conversations and agent_conversation_messages rows remain because no foreign key or account-deletion cleanup applies.

## medium: Paystack transaction audit records cascade away

id: fnd_sig-feat-infra-0973937ca8-84e08f_b4c5b5f7b5
category: data-loss
confidence: medium
triage: risk
status: open
feature: Laravel migrations database/migrations#3 (feat_infra_0973937ca8)

evidence:
- database/migrations/2026_03_25_110228_create_paystack_transactions_table.php:16-18

Paystack transactions are the local record for external payment references, status, response payloads, and webhook idempotency. Cascading them on user or family deletion removes that audit trail and can make later Paystack callbacks or webhooks look like unknown references. This is especially risky because account deletion is a supported user flow in the inspected app.

recommendation:
Preserve external payment records independently from user lifecycle. Add a forward migration to make user_id nullable with nullOnDelete, and consider restrictOnDelete or nullable/anonymized family_id depending on family deletion policy. Scrub PII separately rather than deleting the transaction ledger row.

test analysis:
Paystack tests cover creation, casts, webhook status changes, and double-processing while the transaction exists; profile deletion tests do not seed Paystack transactions or assert audit retention.

suggested regression test:
Add a profile deletion test that creates a PaystackTransaction for the deleting user and asserts the transaction reference/status remains available with the expected anonymized or nulled ownership fields.

minimum fix scope:
Forward migration altering Paystack transaction foreign keys plus account deletion/payment audit regression coverage.

repro:
Create a PaystackTransaction for a user, delete that user account, and then inspect paystack_transactions or send a webhook for the same reference; the transaction row has been deleted by the foreign key cascade.

status: open

- database/migrations/2026_04_20_062943_add_whatsapp_columns_to_users_table.php:15-18

The schema permits multiple users to store the same verified WhatsApp phone because it only adds a normal index. The inspected verification request validates format but not uniqueness, and inbound WhatsApp user lookup resolves by phone to the first verified user. That can attribute inbound messages or reminder actions to the wrong member or family when two accounts verify the same number.

recommendation:
Add a forward migration that deduplicates existing rows and enforces uniqueness for verified WhatsApp phone numbers, ideally with a user-facing validation rule before verification. Because nulls are allowed, a unique index on the normalized E.164 value is sufficient if all stored values remain in the validated format.

test analysis:
The WhatsApp settings tests cover valid format, invalid format, OTP success, and removal, but they do not create two users with the same verified phone or exercise inbound webhook attribution under duplicates.

suggested regression test:
Add a feature test that verifies one user with a phone number, attempts to verify the same number for a second user, and asserts the second verification is rejected and inbound lookup remains unambiguous.

minimum fix scope:
Forward migration for the unique constraint plus WhatsApp verification validation/test coverage.

repro:
Verify the same E.164 WhatsApp number on two different users, then process an inbound webhook from that number; the database allows both verified rows and the lookup has no unambiguous owner.

## low: The day option accepts invalid values and can send the wrong reminder batch

id: fnd_sig-feat-cli-command-1169cff156-_23e1a8d338
category: api-contract
confidence: high
triage: contract-mismatch
status: open
feature: Laravel command contributions:remind (feat_cli-command_1169cff156)

evidence:
- app/Console/Commands/SendContributionReminders.php:12 (SendContributionReminders)
- app/Console/Commands/SendContributionReminders.php:21-23 (SendContributionReminders::handle)
- tests/Feature/Commands/SendContributionRemindersTest.php:18
- tests/Feature/Commands/SendContributionRemindersTest.php:35

The command contract documents only day 25 or 28, but the implementation casts any input and maps all values below 28 to `reminder` and all values 28 or higher to `follow_up`. A typo such as `--day=2`, `--day=99`, or `--day=abc` can therefore dispatch a real notification batch instead of failing fast.

recommendation:
Validate the option before querying contributions. Accept only 25 and 28, emit an error, and return `self::FAILURE` for any other explicit value. If no option is provided, either validate that today is one of the scheduled days or require the option in scheduled usage.

test analysis:
The tests cover only the two valid values, 25 and 28, and do not assert that invalid or non-numeric input is rejected without dispatching notifications.

suggested regression test:
Add a Pest case that runs `contributions:remind --day=99` and `--day=abc`, asserts a failure exit code, and asserts `Notification::assertNothingSent()`.

minimum fix scope:
Add option validation in `SendContributionReminders::handle` before `$type` is derived and before the contribution query runs.

repro:
Run `php artisan contributions:remind --day=99`; the command treats it as `follow_up` and processes current-month unpaid contributions even though 99 is outside the documented option contract.
